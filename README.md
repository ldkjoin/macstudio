# Mac Studio 库存监控

一个运行在本机的 Mac Studio 库存监控工具。它会用后台 Chrome 定时打开苹果中国官网的 Mac Studio 购买页，检查你指定的芯片、CPU、GPU、统一内存和存储组合是否进入可购买流程，并在有货或服务异常时通过 Pushover 通知你。

这个项目的目标很朴素：少刷网页，少焦虑，把“有没有货”这件事交给电脑反复确认。

## 项目初心

有些 Mac Studio 配置不是每天都稳定可买，尤其是高配、定制内存或大容量存储组合。手动刷苹果官网的问题很明显：

- 页面变化不一定直观，可能需要点到购买流程里才能确认。
- 人不可能一天到晚盯着网页。
- 有货窗口可能很短，错过就要继续等。
- 简单请求页面不够可靠，最终还得验证目标配置和购买按钮状态。

所以我做了这个小工具：它不是爬虫平台，也不是抢购脚本，只是一个本地化、可控、可解释的“库存看守”。它会定时检查目标配置，记录每次判断原因，并在真正有信号时提醒你。

## 适合谁

- 想买苹果中国官网指定 Mac Studio 配置的人。
- 不想频繁手动刷新购买页的人。
- 希望检测过程保留历史记录和截图证据的人。
- 已经在使用或愿意配置 Pushover 通知的人。
- 想在自己 Mac 上本地运行，不想把 token 或接收人信息交给第三方服务的人。

## 它会做什么

- 用后台 Chrome 打开苹果中国官网 Mac Studio 购买页。
- 根据页面最终 URL、配置汇总、必选项和“继续”按钮状态判断库存。
- 支持在页面里选择目标型号配置。
- 支持上班时间和下班时间使用不同检测频率。
- 支持 Pushover 普通通知、库存 emergency 强提醒和测试通知。
- 支持多个 Pushover User Key / Group Key 接收人。
- 支持每天定时发送服务状态总结。
- 支持连续检测失败 5 次后发送失败告警，避免服务静默坏掉。
- 保存检测历史、判断原因、通知摘要和通知详情。
- 服务运行期间自动防止 Mac 系统睡眠，避免后台检测中断；屏幕仍按系统设置关闭。

## 它不会做什么

- 不会自动下单。
- 不会绕过苹果官网购买流程。
- 不会上传你的 Pushover token、接收人、数据库或日志。
- 不会把运行时数据提交到 GitHub；`data/`、`logs/`、`.env` 等都在 `.gitignore` 中。

## 工作原理

每次检测会做一轮完整判断：

1. 打开目标 Mac Studio 精确配置 URL。
2. 等待页面完成基础加载。
3. 读取页面标题、最终 URL 和正文内容。
4. 查找“你的新 Mac Studio”配置汇总。
5. 检查目标配置关键词是否都在配置汇总中。
6. 自动选择“不折抵换购”等必选项。
7. 判断购买流程里的“继续”按钮是否可点击。
8. 保存截图和判定原因。
9. 根据结果记录 `in_stock`、`out_of_stock` 或 `error`。
10. 必要时发送 Pushover 通知。

这样做比单纯请求一个 URL 更稳，因为库存判断不只看页面是否打开，还会验证购买流程是否真的能继续。

## 页面使用方式

1. 打开 `http://127.0.0.1:4873`。
2. 在“检测配置”里选择目标 Mac Studio 配置。
3. 配置上班时间、下班时间和对应检测间隔。
4. 在“通知配置”里启用 Pushover。
5. 填写 Pushover Application API Token。
6. 添加一个或多个 User Key / Group Key。
7. 保存配置。
8. 点击启动定时检测。
9. 可以先点“测试 Pushover”确认通知通道可用。

修改配置前需要先停止定时检测。保存后，后续检测会按新配置执行。

## 检测频率

默认配置：

- 上班时间：`09:00-18:00`
- 上班时间检测间隔：每 5 分钟
- 下班时间检测间隔：每 10 分钟

这些都可以在页面里改。对于高配库存监控，我个人更倾向于白天频率高一点，晚上频率低一点，既能及时发现变化，也不会制造太多无意义请求。

## 状态总结

项目支持每天固定时间发送服务状态总结。默认时间为：

```text
09:05
14:05
22:05
```

你可以在页面里调整总结时间，也可以关闭总结。比如只想每天早上确认服务还活着，可以只保留 `09:05`。

总结内容会包含：

- 今日执行次数
- 成功检测次数
- 失败检测次数
- 有货次数
- 无货次数
- 上次总结后的增量统计

## Pushover 配置说明

Pushover 是当前唯一通知通道。

- API Token：在 Pushover 后台创建 Application 后获得。
- User Key：个人账号首页展示的接收 Key。
- Group Key：Pushover Delivery Group 的接收 Key，可像 User Key 一样添加。

通知类型：

- 测试通知：确认 API Token 和接收人配置是否正确。
- 状态总结：按设定时间发送普通通知。
- 有货通知：使用 emergency priority 强提醒。
- 连续失败告警：连续 5 次检测失败后发送一次普通告警。

有货通知支持配置：

- 每个接收人推送条数
- emergency retry 间隔
- emergency expire 时间

截至 2026-05-17，Pushover 官方价格为 30 天试用，之后个人版每个平台 4.99 美元一次性购买；普通账户通过 API 每月可免费发送 10,000 条消息，足够本项目常规使用。

## 连续失败告警

如果检测连续失败 5 次，项目会发一条 Pushover 告警，提醒你检查网络、代理、Chrome 或苹果官网状态。

告警策略：

- 连续失败达到 5 次时发送一次。
- 如果后续继续失败，不会每次都刷屏。
- 只要有一次检测恢复为非错误状态，告警状态会重置。
- 下一轮再次连续失败 5 次时，会重新提醒。

这个机制主要是为了避免服务“看起来还在运行，但实际已经连续报错”的情况。

## 网络与代理

苹果官网、GitHub 或 Pushover 在不同网络环境下可能表现不稳定。如果你使用 Clash、V2Ray、Surge 等代理工具，建议让系统代理保持可用。

当前检测 Chrome 会跟随系统代理，不再强制直连。这样在直连 Apple HTTPS 不稳定时，检测更容易保持正常。

如果检测历史里连续出现类似错误：

```text
page.goto: net::ERR_CONNECTION_CLOSED
```

可以优先检查：

- 系统代理是否开启。
- Chrome 是否能正常访问苹果官网。
- Clash / V2Ray 节点是否可用。
- Pushover 是否能正常发送测试通知。

## 数据与隐私

运行时数据保存在本地：

```text
data/
```

日志保存在：

```text
logs/
```

这些目录不会被 Git 跟踪。你的 Pushover API Token、User Key、检测历史、截图和数据库都只保存在本机。

提交代码前可以用下面命令确认敏感目录没有被跟踪：

```bash
git ls-files data logs .env
```

如果没有输出，说明这些路径没有进入 Git。

## 常见问题

### 为什么显示无货，但苹果官网页面能打开？

能打开页面不等于目标配置可购买。项目会继续验证精确配置 URL、配置汇总和购买流程按钮。如果目标精确配置页回退到通用购买页，或者页面没有出现目标配置汇总，就会判定为无货。

### 为什么要用 Chrome？

苹果购买页是动态页面，只请求 HTML 不够可靠。用 Chrome 可以更接近真实用户打开页面和进入购买流程的行为。

### 为什么修改配置前要停止定时检测？

这样可以避免检测过程中同时修改配置，导致一次检测使用半旧半新的状态。先停、再改、再启动，结果更可解释。

### 有货通知为什么用 emergency？

有货窗口可能很短，普通通知容易被忽略。emergency 会按配置重复提醒，直到确认或过期，更适合真正的库存提醒。

### 可以监控别的产品吗？

当前项目围绕 Mac Studio 配置做了选项、关键词和页面判断。如果要监控其他 Apple 产品，需要扩展配置选项和页面判定逻辑。
